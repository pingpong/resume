# 自己紹介
## 概要
エンジニアとしてECパッケージの開発からキャリアスタートし、主に受託開発を中心に課題のヒアリング・要件定義から保守まで全てを一貫して行う。
その後、テックリードとして自社サービスの立ち上げ、顧客のサポートまで一貫して行う。
直近では10名の開発部のマネージャーとしてPJの管理や技術的なリードに加えて、採用、1on1、評価等のピープルマネージメントを行う。

- サービス開発の全工程を通した開発経験
- テックリードとして技術的な判断の経験
- 10名のチームのマネージメント経験

## 技術スキル
- 同じプロダクトを8年にわたって開発し成長する中での課題と向き合ってきた
- PHP経験10年を軸に、バックエンド・フロント一貫してアプリケーション開発を行ってきた
- AWSを用いたクラウド環境の構築

## ソフトスキル
- オープン：マネジメントする上で、メンバーへ隠し事をせず、言いづらいこと・不都合なことも正直に伝えることを大事にしてきた
- メンバーのサポート：2,3名のメンバーに対して、技術的な目標設定を一緒に行ったり、そこに至る過程を助言してきた
- 責任感：少人数の中で、自分がやるしか無い状況を多々経験してきた。やる以上は、どんな案件でも必ずゴールまで導くことを意識してきた

# 職歴
## 株式会社ソウルウェア (2013/8 ~ )
一人目の正社員、かつエンジニアとして参画しました。10年で組織が2名から26名に成長する過程を、リーダーやEMのポジションで経験しました。　

### 開発部のマネージャーとして (2022/1 ~ )
#### チーム編成
- EM/スクラムマスター 1名
- PO 1名
- エンジニア 10名
- CS/Salesとの連携

#### チームの特徴・課題
会社のビジョンを実現するためには、メンバーが不足していたため、代表と話して、増員計画を立てた。
実際に1年でメンバーが5名から倍になる過程で、チーム内での情報の共有が不足し認識のズレが多く発生してしまった。

#### 使用した技術・バージョン
- PHP5,7/CodeIgniter2/Laravel7,8
- JavaScript/jQuery
- AWS/EC2/ECS/RDS/Aurora/SQS/SES

#### チームでの役割
10名の開発部のマネージャーとして採用、1on1、評価。
技術的な判断やメンバーのサポート。

#### チームの課題と自身が行った意思決定
- 開発のフレームワークがない状況から、チームで開発に取り組めるようにスクラムを導入した。スクラムに関する社内読書会を行いつつ、実際にチーム開発に取り入れた
  - デイリースクラムやプランニングなどのイベントでチーム内で議論が増え改善
- 急拡大する組織内で、チーム内外の信頼関係を築く必要があり、コミュニケーション量を増やす目的で、1on1や社外コーチの導入を進めた。初期は雑談からはじめ、少しずつ仕事への考え方などを相互理解できる状態に引き上げた
- 人数が増え新しいメンバーが両方のサービスを担当する難しさがあったため、チームを製品ごとのチームを分割した
- 代表からは意見交換の活発さなどは評価されつつも、代表が期待するほどの大幅な開発速度向上にはいたっておらず、マネジメントとして十分な成果を挙げられたと思えるほどでもなく難しさを感じた

### 開発チームのリーダーとして (2016/1 ~ 2021/12)
#### チーム編成
- PdM 1名
- エンジニア 5名（リーダーを担当）
- CS/Salesと連携

#### 使用した技術・バージョン
- PHP5,7,8/CodeIgniter2/Laravel5,7,8
- JavaScript/jQuery
- AWS/EC2/RDS/Aurora/SQS/SES/Docker

#### チームでの役割
主力サービスのRepotoneUの開発を担当。各自がバラバラに開発している中で、実装だけでなく、仕様決めから顧客対応までを広く担当。

- Cybozu社のkintone上でPDF/Excelを出力する自社サービスRepotoneUの製品開発
  - ユーザーがPDF/Excelのデザインをブラウザ上でノーコードで行えるサービスの立ち上げから機能追加
  - 顧客からの問い合わせに対して開発として対応するかどうかを判断
  - PDFの郵送サービスや電子契約サービス連携の仕様決めやAPI調査から実装まで
    - 出力されたPDFをAPIを利用して外部サービスと連携
- ソウルウェア内でライセンス発行から請求書の送付業務を自動化
  - プラグインの販売において既存の銀行振込以外の決済方法(Stripe/Paid)の追加

#### チームの課題と自身が行った意思決定
- 全社的に、コードレビューにおいてフォーマットのコメントが増えていたため、フォーマット以外のレビューに集中できるように、PHP Coding Standards Fixerを導入し、コーディング規約を設けた。
- プロダクト内でPHPのバージョンアップが部分的に進める中で、開発者がスムーズに開発できるようにDockerを導入した

#### 成果
- RepotoneU
  - リリース5年間で契約数が平均140%/年で成長
  - ユーザーから競合製品の中で使いやすい・操作がシンプルと評価

# やりたい・目指したいこと
将来的には、EMとしての経験から組織開発・開発チームの構築・チームビルディングをしたいです。
チーム・メンバーの考え方を変え、モチベーションを向上させること、開発者体験を向上させたいです。
それにより、顧客の課題を正しく解決していけるプロダクトを届けられるチームを作りたいです。

これまでは小さな規模の企業内で手探りでスクラムやマネジメントを経験してきたので、小規模な組織であればスムーズにマネジメントができると考えます。
一方、大規模なチームでの経験不足も感じています。
そのような組織でのメンバーとしてのニーズと、小規模組織でのEMとしてのニーズ、それぞれで期待されることがあればお話を聞いてみたいです。
